# M3.2 — Web Admin Dashboard

**Status:** Proposal (required for MVP)
**Scope:** Next.js admin pages (guardscan-api)
**Depends on:** M3.1 (auto-publish at 90% threshold)
**Exit criteria:** Admin can review pending submissions, approve/reject/edit, and manage user-reported inaccuracies from a web UI (not CLI)

---

## Goal

Replace the CLI-based `npm run admin:submissions` workflow with a web dashboard for reviewing submissions and managing user reports. This is critical for MVP because:

1. **Scalability:** CLI tool is fine for N=5–10 pending submissions. At N=100+, a UI is essential.
2. **Audit trail:** Web UI provides better logging of who made what decision and when.
3. **User reports:** New feature (M3.2) to let users flag inaccurate products — needs a web queue.

## Database Schema Changes

### New table: `product_reports`

```sql
create table product_reports (
  id              uuid primary key default gen_random_uuid(),
  product_id      uuid not null references products(id) on delete cascade,
  user_id         text not null,
  issue_type      text not null check (issue_type in ('wrong_ingredients', 'wrong_category', 'missing_ingredient', 'other')),
  issue_description text not null,
  user_photo_url  text,                 -- optional photo from user showing actual product
  status          text not null default 'open' check (status in ('open', 'investigating', 'resolved', 'closed')),
  admin_notes     text,
  reviewed_by     text,                 -- admin user ID
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);

create index on product_reports (product_id, status);
create index on product_reports (user_id, created_at);
create index on product_reports (reviewed_by);
```

### Modified table: `user_submissions`

Add audit trail columns:
```sql
alter table user_submissions add column reviewed_by text;      -- admin user ID (null = auto-published)
alter table user_submissions add column admin_notes text;      -- rejection reason or editorial notes
alter table user_submissions add column decision_at timestamptz; -- when admin decided
```

## API Endpoints

### Admin submission review

```
GET /api/admin/submissions
  Query: ?status=pending|all&confidence_min=70&sort=newest
  Response: {
    submissions: [
      { id, barcode, created_at, confidence, status, submission_count_by_status }
    ],
    total: number,
    cursor?: string
  }

GET /api/admin/submissions/[id]
  Response: {
    submission: {
      id, barcode, user_id, photos, extracted_data (name, brand, category, ingredients),
      confidence, status, created_at
    }
  }

POST /api/admin/submissions/[id]/approve
  Body: { notes?: string }
  Effect: Mark 'published', update reviewed_by + decision_at

POST /api/admin/submissions/[id]/reject
  Body: { reason: string }
  Effect: Mark 'rejected', update admin_notes, reviewed_by, decision_at

POST /api/admin/submissions/[id]/request-edit
  Body: { message: string }
  Effect: Mark 'pending_user_edit', send notification to original submitter
```

### User reporting

```
POST /api/products/[id]/report
  Auth: User (via JWT)
  Body: {
    issue_type: 'wrong_ingredients' | 'wrong_category' | 'missing_ingredient' | 'other',
    description: string,
    photo_uri?: string  // optional Supabase signed URL or upload
  }
  Effect: Insert into product_reports, increment report count

GET /api/admin/reports
  Query: ?status=open|investigating|all&sort=newest
  Response: { reports: [...], total }

GET /api/admin/reports/[id]
  Response: { report + all reports for this product }

POST /api/admin/reports/[id]/resolve
  Body: { status: 'resolved' | 'closed', notes?: string }
  Effect: Update status + reviewed_by + admin_notes

POST /api/admin/reports/[id]/unpublish-product
  Effect: Mark product as 'unpublished' (remains in DB but hidden from scans)
```

## Implementation

### Phase 1: Backend API (4 hours)

**Create endpoints:**
1. `GET /api/admin/submissions` (list pending)
2. `GET /api/admin/submissions/[id]` (detail)
3. `POST /api/admin/submissions/[id]/approve` (publish)
4. `POST /api/admin/submissions/[id]/reject` (reject with reason)
5. `POST /api/products/[id]/report` (user report submission)
6. `GET /api/admin/reports` (list reports)
7. `GET /api/admin/reports/[id]` (report detail)
8. `POST /api/admin/reports/[id]/resolve` (close/investigate)

**Add DB schema:**
- `product_reports` table
- Update `user_submissions` with audit columns

**Auth guard:**
- All `/api/admin/*` routes require `requireAdmin(request)` (check `ADMIN_USER_IDS` env var)
- `/api/products/[id]/report` requires authenticated user

### Phase 2: Frontend UI (6 hours)

**Pages:**
1. `/admin/submissions` — list + filters + bulk actions
2. `/admin/submissions/[id]` — detail view + approve/reject/edit buttons
3. `/admin/reports` — list + status filters
4. `/admin/reports/[id]` — report detail + user photos + action buttons

**Components:**
- `SubmissionCard` — compact submission preview
- `PhotoGrid` — side-by-side front + back
- `IngredientEditor` — inline edit of ingredients
- `ReportDetails` — show all reports for a product + user context

**Styling:**
- Reuse existing theme tokens from `styles/theme`
- Dark mode compatible
- Mobile-friendly (submissions viewed on phone from field)

### Phase 3: User reporting UI (2 hours)

**In `/product/[id]` (Expo app):**
- Add "Report inaccuracy" button at bottom of page
- Modal: select issue type + text input + optional photo
- Submit via `POST /api/products/[id]/report`

**After submission:**
- Toast: "Thanks! We'll investigate this."
- Track reports client-side to prevent duplicate reports from same user

## Approval Thresholds

**M3.1 auto-publish:** Confidence ≥ 90% (stricter than 85%)
- **Guardrails still apply:** name, category, ≥2 ingredients (non-negotiable)
- **Expected rate:** ~25–30% of submissions (vs. 40–50% at 85%)
- **Benefit:** Fewer admin reviews, higher quality baseline

**Manual review (admin dashboard):**
- Submissions with confidence 70–89% (most review load here)
- All rejected submissions (show reason to user)
- Low-confidence submissions (< 70%) shown with warning label

**User reports auto-threshold:**
- ≥5 reports → auto-flag for admin (priority queue)
- ≥10 reports on same issue → recommend unpublish
- Duplicate reports from same user ignored (counted as 1)

## Security

1. **Admin auth:** Middleware checks `ADMIN_USER_IDS` against JWT user ID
2. **Signed URLs:** Photos from Supabase Storage accessed via fresh signed URLs (never public)
3. **Audit trail:** All admin actions logged with `reviewed_by` + `decision_at`
4. **Rate limiting:** User can only report same product once per week (tracked by `user_id` + `product_id`)

---

## Success Metrics

Post-launch, track from logs + dashboard:

1. **Auto-publish rate at 90%** — % of submissions that auto-publish (should drop to ~25–30%)
2. **Admin review time** — average time from pending → approved/rejected (target: < 5 min)
3. **Report volume** — reports per 1000 product scans (baseline TBD)
4. **Report resolution rate** — % of reports closed within 7 days (target: ≥80%)
5. **Product unpublish rate** — % of products unpublished due to reports (should be < 1%)
