# M2.5 — Recommendations Backing API

**Status:** Planning
**Depends on:** M2 (cron ingest — DB must have enough scored products with subcategories)
**Exit criteria:** Expo `getRecommendations` / `getAlternatives` return real data from the backend. `EXPO_PUBLIC_USE_MOCK_API` can be set to `false` for recommendations.

---

## Goal

Replace the Expo app's mock recommendations with real backend endpoints. When a user scans a Poor or Mediocre product, show them better alternatives in the same product subcategory. Two new routes + subcategory inference + scan event tracking.

---

## What Changes

### 1. Subcategory Inference

**New file: `lib/subcategory.ts`**

Pure function that maps product name + category tags to a fixed vocabulary matching the Expo app's `SUBCATEGORY_HINTS`:

```
sunscreen | shave | shampoo | cleanser | deodorant | body_wash
conditioner | moisturizer | drink | snack_bar | cereal | spread
sauce | multivitamin | protein | omega | probiotic
```

`inferSubcategory(name, category, categoryTags?) → string | null`

Called during:
- Normalization (scan-time products)
- Cron ingest (OBF delta, DSLD sync)
- The `after()` cache write in the scan route

Products with no matching subcategory get `null` — they're excluded from recommendations (no peers to cluster against). Adding new subcategories is just adding an entry to the hints array.

### 2. Scan Events Table

**Modified file: `db/schema.ts`**

New `scan_events` table to track which user scanned which product:
```
scan_events
  id          uuid PK
  user_id     text NOT NULL
  product_id  uuid FK → products.id ON DELETE CASCADE
  scanned_at  timestamptz DEFAULT now()
```
Indexes: `(user_id, product_id)`, `(user_id, scanned_at)`

RLS: authenticated users read own rows (`auth.uid()::text = user_id`), service_role full access.

The scan route writes a scan event in the `after()` callback alongside the product cache write. This enables personalized recommendations and future analytics.

### 3. DB Index for Recommendations

**New index on `products`:** `products_subcategory_score_idx` on `(subcategory, score) WHERE subcategory IS NOT NULL`

This makes the alternatives query fast — it can seek directly to the right subcategory and scan by score.

### 4. Alternatives Endpoint

**New file: `app/api/products/[id]/alternatives/route.ts`**

`GET /api/products/:id/alternatives`

1. Look up source product by UUID
2. If no subcategory or no score → return `[]`
3. Query: same subcategory, `score >= source.score + 15` (MIN_SCORE_DELTA), `id != source.id`, order by score DESC, limit 10
4. Reconstruct `ProductAlternative[]` with auto-generated `reason` (e.g. "Scores 23 points higher with fewer concerning ingredients")
5. Cache-Control: `s-maxage=3600, stale-while-revalidate=86400`

### 5. Recommendations Endpoint

**New file: `app/api/recommendations/route.ts`**

`GET /api/recommendations?category=<optional>`

1. Get user_id from auth context
2. Query `scan_events` joined to `products`:
   - Only Poor/Mediocre (score < 60)
   - Must have subcategory
   - Deduplicated by product (most recent scan wins)
3. For each scanned product, find the single best alternative (highest score in same subcategory, delta >= 15)
4. Return `RecommendationPair[]` ordered by `scanned_at` DESC, limit 20
5. Optional `category` filter narrows to food or grooming only

This is a complex join implemented as a raw SQL query via Drizzle's `db.execute(sql\`...\`)`.

### 6. Alternatives in ScanResult

**Modified file: `app/api/products/scan/[barcode]/route.ts`**

After scoring, if DB is configured and product has a subcategory + score:
- Query top 3 alternatives inline (before sending response)
- Populate `ScanResult.alternatives` (currently hardcoded `[]`)

This runs inline (not in `after()`) because the client needs the data in the response.

### 7. Subcategory in Normalization + Ingest

**Modified files:** `lib/normalize.ts`, `lib/cron/ingest-helpers.ts`

All normalization paths call `inferSubcategory()` and include the result in the product. The `upsertProduct()` helper writes `subcategory` to the DB.

## Contract with Expo Client

Per charter §13.4:
- `MIN_SCORE_DELTA = 15` — defined in `lib/scoring/constants.ts`, mirrored from Expo's `constants/Recommendations.ts`. Never forked.
- `RecommendationPair` and `ProductAlternative` types match `types/guardscan.ts` (shared between API and Expo)
- Subcategory vocabulary matches Expo's `SUBCATEGORY_HINTS` (currently in `lib/api.mock.ts`, will move to `constants/` when mock is retired)
- `RECS_TAB_MAX_PAIRS = 20` and `STRIP_MAX_ALTERNATIVES = 3` limits match the Expo constants

## Verification

1. **Subcategories populated**: After ingest, `SELECT subcategory, count(*) FROM products WHERE subcategory IS NOT NULL GROUP BY subcategory` shows multiple subcategories with products
2. **Alternatives work**: Given a low-scoring product, `GET /api/products/{id}/alternatives` returns products 15+ points higher in the same subcategory
3. **High-score products**: A product scoring 85+ returns empty alternatives (no product can beat it by 15+)
4. **Recommendations work**: After scanning some Poor products, `GET /api/recommendations` returns valid `RecommendationPair[]`
5. **ScanResult.alternatives populated**: Scanning a product that has DB alternatives returns them in the response
6. **MIN_SCORE_DELTA enforced**: No alternative has a score delta less than 15
7. **Category filter**: `GET /api/recommendations?category=grooming` only returns grooming pairs
